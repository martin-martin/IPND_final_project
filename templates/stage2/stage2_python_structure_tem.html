What is a ...?

  1. computer: it's the universal machine. it's like a machine, but we can make it do anything (that we manage to write a program for)
  2. program: is what tells the compter which steps to take to achieve what we want.
  3. precise sequence of steps: a computer acts logical, and needs good commands.
  4. computation: the act of calculating stuff to achieve results
  5. high-level language: code that we write, that isn't exactly how the computer would understand it
  6. input: the information we give the computer to work with
  7. interpreter: a program (e.g. python) that translates our written stuff into computer speach.

Computer Language

General

Ambiguity in languages: we can't rely that the computer would interpret something correctly. we have to be sure it will do what we want it to do. for this the instructions have to be written unambiguously - and this doesn't work in natural languages.

Verbosity in languages: there's lots and lots of words to say many things in detail, and also lots and lots of words to say basically nothing. sadly it's more often like the latter. Well... computers would get totally confused by this, as also humans do. But humans don't admit it and act like it's awesome cool to know many more words than the person they are talking to. The amount of words has to be smaller when speaking to a computer, so that the meaning of what is trying to be said can and will get transmitted more clearly. sweet.

Grammar

<non-terminal> --> replacement

There's this  thing  about the (context-free) form of a language. I'm not sure how much I really understand it, but it feels  very logical and exciting !

I understand it a bit like references and assignement statements. nt -> rp can be a reference, if rp is another nt, nt -> te is an assignment statment. once the assignment statment has happened, that original nt is now defined and thus a <terminal>.

a  recursive definition  is a function that refers to itself. in programming languages we can do this, because of how the grammar is constructed.

python grammar

the grammar rules:

  • <Expression> --> <Expression> Operator <Expression>
  • <Expression> --> Number
  • <Expression> --> ( <Expression> )

and then you just fill in:

  • Operator can be e.g.  + - * / %  ...
  • Number can be any number
  • Since <Expression> can be substituted with another <Expression>, large constructs can be made from this simple rule

So schmall so beautiful! : D

Some Notes about Python and HTML

Why are there double and single quote options in python? Because when extracting from HTML, there's often double quotes that are in-string! And on the other hand in english there's often a single quote (an apostrophe) within written language. Having only one option would close the string at a point that is actually in-string.

Triple quotes (either """ or ''') can be used to print multi-line strings. (but don't use it as a comment! In python that's reserved for docstrings :)

It's better to use negative indexing, when addressing HTML text through python. For one because it's easier to count short distances, for the other because certain elements are repetitive in HTML, therefore the same structure can be easily re-used and applied to any lenght of string! Saves a lot of work :)
